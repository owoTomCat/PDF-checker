# 页面定位批次漏页自动纠正设计

## 背景

2026-07-19 的线上任务中，8 页 PDF 在第一批 6 页完成浏览器渲染后，`POST /api/audit/layout` 返回 502。百炼返回的是符合 `LayoutBatchSchema` 的 JSON，但 `pages` 没有覆盖请求中的全部页码。当前代码在百炼客户端的两次模型尝试结束后，才由 API 路由比较请求页码与返回页码，因此这种“结构合法但漏页”的输出不会触发已有的模型纠正重试。

同一时段并行处理的另一个 PDF 完成了布局、证据识别、URL 复核、表格提取、关联和最终规则计算，说明服务、API Key 和后续业务链路正常。

## 目标

- 每个布局批次必须为请求中的每一页返回且只返回一个页面对象，允许某页的 `regions` 为空。
- 第一次模型输出漏页、增加页或重复页时，利用现有的第二次模型尝试进行纠正。
- 第二次输出仍不完整时继续返回稳定的 `INVALID_MODEL_OUTPUT`，不得伪造空白页面或静默跳过证据。
- 不增加模型最大尝试次数，不记录原始模型输出，不放宽现有区域结构与安全校验。

## 方案比较

1. **在百炼客户端内部校验页码覆盖并纠正重试（采用）**：覆盖校验与模型输出解析处于同一重试边界，能够复用已有的纠正提示和两次尝试限制。
2. 在浏览器端重试整个 `/api/audit/layout` 请求：需要重新上传整批页面图，网络和模型成本更高，也会在客户端重复服务端职责。
3. 为缺失页补空对象后继续：会把“模型漏页”误判为“页面没有证据”，违反严格核验规则。

## 设计

### 动态布局输出契约

`locateRegions` 根据本次输入生成带上下文的布局输出 schema。该 schema 在现有 `LayoutBatchSchema` 之上校验：

- 返回页码集合与输入 `pages[].pageNumber` 集合完全相等；
- 每个预期页码恰好出现一次；
- 不允许模型返回本批次之外的页码。

空区域页面应返回 `{ pageNumber, regions: [], warnings, confidence }`，而不是省略整个页面。

### 重试行为

动态 schema 校验失败时，沿用 `INVALID_MODEL_OUTPUT` 的可重试路径。第二次请求追加布局专用纠正说明，明确列出预期页码，并要求每页恰好返回一次，即使没有区域也必须返回空 `regions`。

最大尝试次数保持为 2：

1. 第一次返回完整，直接成功；
2. 第一次漏页等不完整、第二次完整，成功；
3. 两次都不完整，返回 502 和稳定的中文错误，不泄露百炼原始响应。

API 路由保留现有集合相等检查作为纵深防御，但正常情况下不再由它首次发现漏页。

## 数据流

页面批次图像 → 百炼第一次布局识别 → 动态 schema 校验页码覆盖 → 不完整时追加预期页码纠正提示 → 百炼第二次布局识别 → 再次校验 → 完整结果进入裁剪和后续识别；否则返回稳定失败。

## 测试与验收

- 客户端测试：第一次漏掉一个请求页、第二次完整时共调用两次并成功。
- 客户端测试：两次均漏页时只调用两次并抛出 `INVALID_MODEL_OUTPUT`。
- 客户端测试：第二次请求包含预期页码和“空区域页也必须返回”的纠正说明。
- 现有布局 schema、API、完整流水线测试全部保持通过。
- `npm test`、`npm run typecheck`、`npm run lint` 和 Node 24 生产构建通过。
- 使用触发问题的 8 页真实 PDF 重新处理，确认第一批不会因一次漏页直接失败，并核对所有后续 API 状态与最终页面结果。

## 非目标

- 不改变批次大小、并行文件数或模型型号。
- 不把缺页降级为人工复核结果。
- 不保存 PDF、页面图或模型原始响应。
