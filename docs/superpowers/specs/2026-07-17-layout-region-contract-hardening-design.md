# 布局区域契约加固设计

## 背景

真实 PDF `（2026）苏0281民初7549号-3-外网溯源结果报告.pdf` 在布局阶段得到 HTTP 200，但模型把独立权利图和对比图误标为 `rights_screenshot`，同时返回了空 `resultIndex`。该结果通过 `LayoutBatchSchema` 后进入证据识别阶段，最终被该阶段的元数据校验拒绝，页面显示“证据区域元数据重复、越界或父子关系不完整”。

## 目标

让布局阶段只接受下游可以确定性处理的区域元数据，并明确区分真实网页截图、独立权利图、对比图、案件信息表和结果汇总表。无效模型输出应在布局阶段触发既有的一次 JSON 纠正重试，而不是延迟到证据阶段失败。

## 方案

1. 加强 `LAYOUT_SYSTEM_PROMPT`：
   - `rights_screenshot` 仅表示与某个结果对应、包含网页或浏览器界面的完整外网截图。
   - 独立权利图、图片对比区域、封面装饰图、报告标题和案件信息表不得标为 `rights_screenshot`。
   - `summary_table` 仅表示包含网络来源、发布者、发布时间、URL 等结果行的明细汇总表；案件基本信息表不属于该类型。
   - 截图必须同时提供正整数 `rightsImageIndex` 和 `resultIndex`；不能可靠确定时省略区域并写入 warning。
2. 加强 `PageLayoutSchema` 的类型不变量：
   - `certificate`、`summary_table`：无父区域，两个索引均为 `null`。
   - `rights_screenshot`：无父区域，两个索引均为正整数。
   - `address_bar`：父区域必须是同页 `rights_screenshot`，且两个索引与父区域一致。
3. 保留证据 API 的严格拒绝策略，不在客户端静默丢弃无效区域，也不根据内容猜测索引。

## 数据流

模型布局输出 → `LayoutBatchSchema` 校验 → 无效时由百炼客户端追加纠正提示并重试一次 → 有效布局才进入裁剪和证据识别。

## 测试与验收

- 契约测试覆盖每种区域的索引和父子关系。
- 客户端测试证明缺失截图索引的模型输出会触发纠正重试。
- 提示词测试覆盖排除项与汇总表定义。
- 全量 `npm test` 和 `npm run typecheck` 通过。
- 用上述 4 页真实 PDF 在浏览器中重新处理，流程进入最终报告，不再出现证据区域元数据错误。
